    # **`gcTime`**과 **`staleTime`**의 차이점에 대해 정리해보세요 🍠
    
    <aside>
    🍠
    
    gcTime과 staleTime은 Tanstack Query에 있어서 정말 중요하게 알고 넘어가야 하는 부분이에요.
    
    **위에서 제가 설명해드렸지만 gcTime**과 **staleTime**의 개념을 다시 정리해주시고, 두 값을 어떤 식으로 설정하면 캐싱 전략에 유리한지 설명해주세요!

    - `gcTime`은 무엇인가요? 🍠
        
        “**메모리 청소**”를 담당한다
        
        **1️⃣ gcTime이란?**
        
        - **정의** : 컴포넌트가 언마운트(화면에서 사라짐)되어 해당 쿼리가 **사용되지 않는 (inactive) 상태**가 되었을 때, 캐시 데이터를 메모리에 얼마나 오래 보관할지 결정하는 시간이다.
        - **기본 값** : 5분
        - **이전 명칭** : `cacheTime`이라는 이름이었다. 역할이 더 명확하게 `gcTime`으로 변경되었다.
        
        **2️⃣ gcTime의 동작 원리**
        
        1. 사용자가 특정 페이지를 떠나면 해당 데이터는 `inactive` 상태가 된다.
        2. 이때부터 `gcTime` 타이머가 돌아가기 시작한다.
        3. 타이머 시간에 따라
            1. **타이머 종료 전** 다시 해당 페이지로 돌아온다면 ➡️ 캐시된 데이터를 즉시 보여준다.
            2. **타이머 종료 후** 해당 페이지로 돌아온다면 ➡️ 캐시가 삭제되었으므로 로딩 스피너 (`isPending`)를 다시 보게 된다.
        
         **3️⃣ 예시 코드**
        
        ```tsx
        const { data } = useQuery({
          queryKey: ['todos'],
          queryFn: fetchTodos,
          // 1분 동안은 데이터가 신선하다고 간주 (네트워크 요청 안 함)
          staleTime: 1000 * 60, 
          // 컴포넌트가 사라져도 10분 동안은 메모리에 저장 (다시 돌아왔을 때 깜빡임 방지)
          gcTime: 1000 * 60 * 10, 
        });
        ```
        
        **4️⃣ 유의사항**
        
        - **`gcTime`**은 **`staleTime`**과 독립적이다. `staleTime`이 아무리 길어도 `gcTime`이 지나 버리면 캐시는 삭제된다
        - **사용자 경험(UX) 최적화** : 사용자가 자주 왔다 갔다 하는 페이지라면 `gcTime`을 넉넉하게 잡아서 다시 들어와도 로딩 없이 화면이 바로 보이게 할 수 있다
        - **메모리 관리** : 데이터 양이 너무 많거나 메모리가 부족한 환경이라면 `gcTime`을 길게 잡는게 좋지 않다.
        
    - `staleTime`은 무엇인가요? 🍠
        
        “**데이터가 언제까지 신선한지(Fresh)**”를 결정하는 타임이다.
        
        **1️⃣ staleTime이란?**
        
        - **정의** : 서버에서 가져온 데이터가 **신선한 (Fresh) 상태**로 유지되는 시간이다.
        - **상태 변화** : 이 시간이 지나면 데이터는 **낡은 (Stale)** 상태가 된다
        - **기본 값** : 0 (데이터를 가져오자마자 바로 Stale 상태가 된다)
        
        2️⃣ **왜 사용하는가?**
        
        - **불필요한 네트워크 요청 감소** : `staleTime`이 1분이면 사용자가 컴포넌트를 몇번을 껐다 켜도 1분 안에 서버에 다시 물어보지 않고 캐시된 데이터를 그대로 쓴다.
        - **사용자 경험(UX) 향상** : 화면이 깜빡이지 않고 즉각적으로 데이터를 보여줄 수 있다.
        
        **3️⃣ 데이터가 Stale 해지면 생기는 일은?**
        
        데이터가 낡았다고 바로 사라지지는 않는다. 아래와 같은 **트리거가 발생했을 때** TankStack Query가 서버에서 데이터를 다시 요청(Refetch)하게 된다.
        
        1. 새로운 Query instance가 mount 될 때 (예: 페이지 이동 등)
        2. 브라우저 창에 다시 포커스가 왔을 때 (`refetchOnWindowFocus`)
        3. 네트워크가 끊겼다가 다시 연결될 때 (`refetchOnReconnect`)
        4. 설정한 `refetchInterval` 시간이 되었을 때
        
        **4️⃣ 실제 코드 예시**
        
        ```tsx
        const { data } = useQuery({
          queryKey: ['todos'],
          queryFn: fetchTodos,
          // 30초 동안은 '신선'하다고 봐줘! (30초간 재요청 금지)
          staleTime: 1000 * 30, 
        });
        ```
        
        5️⃣ **디버그**
        
        - **초록색(Fresh):** `staleTime`이 아직 안 끝난 상태
        - **노란색(Stale):** `staleTime`이 끝나서 트리거만 주면 바로 서버에 물어볼 대기하는 상태
        
        **6️⃣ gcTime vs staleTime 비교**
        
        | **비교 항목** | **staleTime (신선도)** | **gcTime (메모리 관리)** |
        | --- | --- | --- |
        | **핵심 질문** | "언제 서버에 다시 물어볼까?" | "언제 캐시를 아예 지울까?" |
        | **관심 대상** | **네트워크 비용**  | **메모리 자원** |
        | **상태 전환** | Fresh ➡️ **Stale** | Inactive ➡️ **삭제(Deleted)** |
        | **영향력** | 데이터가 신선하면 요청을 안 보냄 | 데이터가 삭제되면 로딩창을 봐야 함 |
        | **기본값** | 0 | 5분 (300,000ms) |
        | **역할** | 재요청(Refetch)을 보낼지 결정 | 메모리에서 데이터를 아예 지울지 결정 |
        | **권장값** | 보통 gcTime보다 짧게 설정 | 보통 staleTime보다 길게 설정 |
    - 두 값을 어떤 식으로 설정하여야 `캐싱 전략에 유리`한가요? 🍠
        
        **1️⃣ 기본 규칙**
        
        > **staleTime < gcTime**
        > 
        
        `gcTime`은 항상 `staleTime`보다 길어야 한다. 만약 `staleTime`이 `gcTime`보다 길면 데이터가 아직 fresh 함에도 불구하고 메모리에서 삭제되어 로딩창을 다시 봐야한다.
        
        **2️⃣ 데이터 성격에 따른 3가지 경우**
        
        **✔️ Case A : 정적 데이터**
        
        - **예시** : 설정 값, 사용자 프로필, 공지 사항, 카테고리 목록 …
        - **전략** : `staleTime`을 넉넉히 잡아 서버의 부하를 줄인다
        - **효과** : 사용자가 페이지를 이동해도 서버 요청이 거의 발생하지 않아 매우 빠르게 느껴진다.
        
        **✔️ Case B : 실시간성 데이터**
        
        - **예시** : 주식 시세, 실시간 채팅 목록, 스포츠 경기 스코어
        - **전략** : `staleTime`을 짧게 주거나 기본값(0)을 유지하여 항상 최신 상태를 보장한다
        - **효과** : 사용자가 화면을 볼 때마다 또는 포커스를 맞출 때마다 새로운 데이터를 가져와 정확한 정보를 제공한다.
        
        **✔️ Case C : 일반적인 데이터**
        
        - **예시** : 할 일 목록(Todo), 피드 포스트, 상품 리스트
        - **전략** : 사용자 경험을 위해 약 30초 ~ 1분 정도의 fresh를 유지한다.
        - **효과** : 너무 잦은 새로고침은 방지하면서 적절한 타이밍에 데이터를 동기화한다.
        
        **3️⃣ UX를 극대화**
        
        사용자가 페이지를 나갔다가 **1분 뒤에 돌아오는 상황**
        
        1. **staleTime 0, gcTime 5분:** 들어가면 `inactive` 캐시가 있으니 옛날 데이터를 보여주지만, 동시에 'Stale'하므로 배경에서 즉시 서버 요청을 보낸다. (데이터가 바뀌면 화면이 깜빡하며 업데이트 됨)
        2. **staleTime 1분, gcTime 5분:** 1분 안에 돌아왔다면 서버 요청 없이 캐시된 데이터를 그대로 쓴다. (깜빡임 전혀 없음)
        
        **✅  요약**
        
        - 서버 트래픽을 아끼고 싶다면 ➡️ `staleTime`을 늘리기
        - 사용자가 다시 돌아왔을 때 로딩바를 안 보여주고 싶다면 ➡️ `gcTime`을 늘리기
        - 항상 최신 데이터를 보여줘야 한다면 ➡️ `staleTime`을 `0`으로 두기

        **❓ 무조건 캐시만 보여주면 안되는 이유는?**
        
        - **데이터 불일치** : 사용자가 할 일을 추가하거나 수정하는 작업을 수행했음에도 불구하고 화면이 읽기용 캐시 데이터만 보여준다면 사용자는 시스템이 내 요청을 무시했다고 오해하거나 오류가 발생했다고 생각하게 된다.
        - **신뢰도 하락** : 실시간성이 중요한 서비스에서 과거의 데이터를 보여주는 것은 서비스의 신뢰도를 떨어뜨리는 요인이 된다.

    ```tsx
    const queryClient = new QueryClient({
      defaultOptions: {
        queries: {
          staleTime: 1000 * 60 * 5, // 5분

        },

          retry: false,
        },
      },
    });
    ```
    
    **Q1. `staleTime: 5분`은 무슨 뜻인가요?**
    
    **Q2. `gcTime: 6분`은 `staleTime`과 뭐가 다른가요?
    
    Q3. `mutations: { retry: false }`는 왜 설정했을까요?**

        <aside>
        ❓
        
        답변
        
        **❓ `staleTime : 5분` 은 무슨 뜻인가?** 
        
        데이터가 서버에서 가져온 후, 데이터가 신선한(Fresh) 상태로 유지되는 시간이 5분이라는 뜻이다.
        
        5분 동안은 화면을 나갔다 들어와도 다시 마운트 되어도 서버에 새로운 요청을 보내지 않고 캐시된 데이터를 사용한다. 5분이 지나면 데이터는 낡은(Stale) 상태가 되고 이후에는 해당 데이터를 필요로 하는 시점에 서버에서 다시 불러온다.
        
        **❓ `gcTime : 6분`은 `staleTime`과 뭐가 다른가?**
        
        `staleTime`이 재사용 여부를 결정한다면, `gcTime`은 데이터가 메모리에 남아있는 시간을 결정한다.
        
        - `staleTime`은 유효기간과 같아서 이 시간이 지나면 다시 가져와야한다고 판단하는 기준이다.
        - `gcTime`은 컴포넌트가 언마운트되어 해당 데이터를 아무도 쓰지 않을 때, 메모리에서 완전히 삭제하기 까지 기다리는 시간이다. 보통 캐시를 효율적으로 쓰기 위해 **`staleTime`보다 길게 설정**한다.
        
        **❓ `mutations: { retry: false }`는 왜 설정 했을까?**
        
        데이터를 생성, 수정, 삭제하는 `mutation` 작업을 실패했을 때 자동으로 다시 시도하지 않도록 설정한 것이다.
        
        - Query (조회)는 데이터를 다시 읽어오면 되지만, 수정이나 삭제 (Mutation)은 네트워크 오류 등으로 실패 했을 때 무작정 다시 시도하면 데이터가 중복적으로 생성되거나 의도치 않은 부작용 (Side Effect)가 생길 수 있다
        - 중요한 변경 작업은 에러를 확인하고 처리하거나, 사용자가 다시 시도하게 만드는게 안전하기에 재시도 기능을 끄는 경우가 있다.
        </aside>

- **미션 5 - enabled** **속성의 중요성**
    
    <aside>
    ❓
    
    `useQuery`에는 **"지금은 요청하지 마"**라고 말할 수 있는 `enabled` 옵션이 있어요. 아래 예제를 보고 질문에 답해보세요.
    
    ```tsx
    function UserProfile() {
      const [userId, setUserId] = useState(null);
    
      const { data: profile } = useQuery({
        queryKey: ['user', userId],
        queryFn: () => fetch(`/api/users/${userId}`).then(res => res.json()),
        enabled: userId !== null,  // ← userId가 있을 때만 요청!
      });
    
      return (
        <>
          <button onClick={() => setUserId('abc123')}>프로필 보기</button>
          {profile && <p>{profile.name}</p>}
        </>
      );
    }
    ```
    
    **Q1.** `enabled: userId !== null`이 없으면 어떤 문제가 생길까요?
    
    **Q2. 실제로 지금 우리가 보고있는 “아이두” 서비스에서는 이런게 어디에 쓰일 수 있을 것 이라고 생각하시나요? 모든 화면들을 보면서 최대한 많이 찾아주세요.**
    
    </aside>
    
    - 답변
        
        <aside>
        ❓
        
        답변
        
         **❓`enabled: userId !== null` 이 없으면 생기는 문제는?**
        
        잘못된 주소로 API 요청을 보내서 404 에러가 발생할 수 있다.
        
        `useQuery`는 기본적으로 컴포넌트가 마운트되자마자 실행된다. `userId`가 초기값인 `null`인 상태에서 `enabled` 옵션이 없다면, 서버에 `/api/users/null`이라는 주소로 요청을 보내게 된다.
        
        → 이는 불필요한 네트워크 트래픽을 발생시킬 뿐만 아니라, 서버 콘솔에 에러 로그를 남기고 클라이언트에서는 유효하지 않은 데이터를 처리하려다 화면이 멈추는 문제를 일으킬 수 있다.
        
        **❓ 실제 서비스에서 어디에 쓰일 수 있을까?**
        
        - **날짜별 상세 할일 조회** : 메인 화면의 캘린더에서 특정 날짜를 클릭했을 때, 그 날짜에 해당하는 상세할 일들이 뜬다. `enabled: !!selectedDate` (날짜가 클릭되어 `selectedDate` 상태값이 존재할 때만 요청)
        - **검색창의 유저 검색** : 친구를 추가하기 위해 아이디를 검색할 때, 검색어의 길이를 제한할 수 있다. `enabled: searchTerm.length >= 2` (검색어가 짧을 때는 API 호출을 막아 서버 부하 방지)
        - **카테고리 관리** : 할일 목록의 카데고리를 수정할때 카테고리의 기존 설정값들을 가져 와야 한다. `enabled: isOpen && !!categoryId` (수정할 카테고리 ID가 확실히 있을 때만 호출)
        - **로그인 여부에 따른 호출** : 유저의 할 일 목록 같은 정보는 로그인한 사용자만 볼 수 있다. `accessToken`이 없을 때는 아예 요청을 보내지 않도록 `enabled: !!accessToken` 처리가 필요하다

        **❓ 위치 정보가 없는 상태에서 불필요한 날씨 요청을 방지하는 방법은?**
        
        React Query의 enabled 옵션을 활용한다. 위치 정보가 확인되지 않을때는 `enabled: !!coords` 설정을 추가한다. 좌표값이 있을때만 활성화 된다. 이런 처리를 통해 승인하거나 위치 정보를 받아오기 전에 쿼리 함수 자체가 실행되지 않으므로 불필요한 서버 요청과 비용 발생을 차단할 수 있다.
        
        **❓ 날씨 데이터의 적절한 `staleTime` 설정은 어떻게 해야할까?**
        
        - **똑같이 5분을 적용할 경우** : 기상청 데이터는 3시간 마다 바뀌는데 5분마다 계속 서버에게 물어보는 꼴이 된다. 변하지 않는 데이터를 위해 네트워크 자원을 낭비하게 된다.
        - **3시간으로 설정할 경우** : 기사청은 이미 데이터를 업데이트 했는데 사용자가 로그인한 시점에 따라 서비스에서는 신선한 데이터라고 생각하여 옛 데이터를 계속 보여줄 위험이 있다.
        - **적절한 설정은?** : 30분 ~ 1시간 정도가 적당할 것 같다. 기상청의 갱신 주기를 고려하되 사용자가 너무 오래된 정보를 보지 않도록 적절한 시간을 설정해야한다.
        
        **❓ 위치가 변경되었을 때 이전 지역의 날씨가 보이면 어떻게 해결할까?**
        
        Query Key에 위치 정보를 포함 시키면 된다. Query Key를 `[’weather’, location]`과 같이 배열 형태로 작성한다. 위치가 서울에서 부산으로 바뀌는 순간 Query key 자체가 변하게 된다. React Query는 이를 서로 다른 데이터로 인식하여 즉시 부산의 날씨를 새롭게 호출하게 된다.
