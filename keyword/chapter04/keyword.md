    # 📚 블로그 읽고 세션과 토큰 정리해보기
    
    ---
    
    [[🔐인증] 2. 세션 vs 토큰: 사용자의 상태를 어디에 저장할까](https://velog.io/@cucumtomatobean/dtd9x1yh)
    
    - 세션 방식과 토큰 방식의 가장 큰 차이는 무엇인가요? 🍠
        
        세션과 토큰 방식의 가장 큰 차이는 **"사용자의 상태 정보를 어디에 저장하는지"**이다.
        
        **1️⃣ 인증 상태의 저장 위치**
        
        **세션 방식 (Stateful)** : 실제 사용자 정보를 서버에 저장하고 클라이언트에는 정보를 찾을 수 있는 키 (sessionID)만 전달한다. 즉, 서버가 사용자의 상태를 기억하고 있는 상태유지 방식이다.
        
        **토큰 방식 (Stateless)** : 사용자 정보를 토큰 자체에 포함시켜 클라이언트가 직접 보관하고 전송하는 방식이다. 즉, 서버는 사용자를 기억하지 않는 무상태 방식이다.
        
        **2️⃣ 인증 과정의 메커니즘**
        
        | **구분** | **세션 (Session)** | **토큰 (Token/JWT)** |
        | --- | --- | --- |
        | **로그인 성공 시** | 서버가 **세션 ID**를 생성하고 DB에 저장함 | 서버가 유저 정보를 담은 **JWT**를 생성해 전달함 |
        | **전달 수단** | 브라우저의 쿠키(Cookie)에 세션 ID를 담음 | HTTP 헤더(Authorization)에 토큰을 실어 보냄 |
        | **서버의 확인** | 들어온 세션 ID가 **DB에 있는지** 뒤져봄 | 들어온 토큰의 **서명(Signature)이 맞는지** 계산해봄 |
        
        ****
        
    - 세션을 분산 환경에서 관리할 때 사용할 수 있는 세 가지 전략(Sticky Session, Session Replication, Centralized Session Store)의 특징을 정리해주세요. 🍠
        
        **분산 환경에서 세션을 공유하는 방법**
        
        https://velog.io/@leo4study/%EB%B6%84%EC%82%B0-%EC%8B%9C%EC%8A%A4%ED%85%9C%EC%97%90%EC%84%9C-%EC%8A%A4%ED%8B%B0%ED%82%A4-%EC%84%B8%EC%85%98%EA%B3%BC-%EC%84%B8%EC%85%98-%EA%B4%80%EB%A6%AC-%EC%9D%B4%ED%95%B4
        
        **🔷 Sticky Session (세션 고정)**
        
        사용자의 요청을 항상 같은 서버로 보내는 방식이다. 
        
        - **특징** : 로드 밸런서(전달자)가 유저의 첫 요청을 보고 특정 서버로 보냈다면, 이후의 모든 요청도 무조건 같은 서버로 보낸다. 보통 쿠키나 IP 주소를 기준으로 고정한다.
        - **장점** : 서버들끼리 세션 데이터를 주고 받을 필요가 없어서 구현이 간단하다
        - **단점** :  서버 장애 시 해당 서버의 모든 세션이 사라진다. 특정 서버에만 유저가 몰리면 **부하 불균형**이 생길 수 있다.
        
        **🔷 Session Replication (세션 복제)**
        
        모든 서버가 세션 정보를 공유하는 방식이다. 
        
        - **특징** : 유저가 어떤 서버에서 로그인을 하든, 세션 정보를 다른 모든 서버에 즉시 복사(broadcasting)한다. 모든 서버가 동일한 세션 정보를 가지게 된다.
        - **장점** : 서버가 죽어도 다른 서버에 이미 정보가 다 있기 때문에 서비스가 매우 안정적이다. 유저는 어떤 서버로 접속해도 로그인이 유지된다.
        - **단점** : 서버가 늘어날수록 복제해야 할 데이터가 급격히 많아져 **메모리 낭비**가 심해지고, 데이터 동기화 과정에서 **네트워크 부하**가 크게 발생한다.
        
        **🔷 Centralized Session Store (중앙 세션 저장소)**
        
        별도 외부 저장소(Redis 나 Memcached)에 모든 세션을 보관하는 방식이다. 
        
        - **특징** : 모든 서버는 요청이 올 때마다 중앙 저장소에서 세션을 확인한다. 가장 많이 사용하고 있는 방식이다.
        - **장점** : 서버가 늘어나도 저장소만 바라보면 되므로 **확장성**이 뛰어나다.
        - **단점** : 중앙 저장소 자체가 죽으면 모든 서비스의 인증이 마비된다. 외부 저장소와 통신해야 하므로 서버 내부 메모리를 쓰는 것보다 지연 시간이 생길 수 있다.
        
        | **전략** | **핵심 개념** | **단점** |
        | --- | --- | --- |
        | **세션 고정** | 특정 서버 고정 | 서버 장애 시 세션 유실 |
        | **세션 복제** | 전 서버 데이터 복제 | 메모리 및 네트워크 부하 |
        | **중앙 세션 저장소** | 외부 통합 저장소 | 저장소 장애 시 전체 마비 |
    - JWT(Json Web Token)의 장점과 단점을 각각 설명해주세요. 🍠
        
        #### 🔷 **장점**
        
        - **확장성** : 서버가 세션 상태를 저장하지 않는 무상태(Stateless) 방식이다. JWT는 독립적인 구조를 가지고 있어서 서버 상태를 유지할 필요 없이 공통된 ‘비밀 키’만 있다면 확장 가능하다.
        - **정보 자가 포함** : 토큰 자체에 필요한 정보를 포함하고 있어서 인증을 위해 매번 DB 조회를 하지 않아도 되므로 서버의 부하가 감소한다.
        - **범용성** : 웹, 쿠키를 사용하기 어려운 모바일 앱, API 등 다양한 환경에서 사용 가능하다.
        - **보안성** : 서명된 토큰은 변조 방지 기능이 있어서 보안성이 높다.
        - **서버 자원 절약:** 유저가 매우 많아도 서버 메모리에 세션 정보를 쌓아둘 필요가 없다. 서버는 토큰이 들어올 때만 검증하면 되므로 부하가 적다
        
        #### 🔷 단점
        
        - **토큰 크기** : 세션은 짧은 세션 ID만 주고받지만, JWT는 유저 정보가 들어있어 문자열이 길다. 따라서 매 요청마다 큰 토큰을 주고받으면 네트워크 대역폭 낭비가 생길 수 있다.
        - **만료 시간** : 만료된 토큰을 무효화하기 어려워 보안 취약점이 생길 수 있다
        - **복잡성** : JWT를 처리하기 위한 로직이 복잡할 수 있으며, 구현이 어려울 수 있다.
        - **서버 측 검증 필요** : 토큰의 유효성을 검증하기 위해 서버 측에서 추가적인 작업이 필요하다
        - **통제 불가** : 한번 발급하면 통제가 불가 하다. 토큰이 해킹 당하거나 유저가 로그아웃을 해도 이미 발급 된 토큰은 만료 시간이 될 때까지 서버가 강제로 뺏어올 수 없다
        - **데이터의 노출:** JWT의 Payload 부분은 단순한 Base64 인코딩일 뿐이라서 누구나 열어볼 수 있다.
        - **수정의 어려움:** 토큰 내부에 담긴 유저의 권한이나 이름이 변경되어도, 이미 발급된 토큰에는 반영되지 않는다. 유저가 다시 로그인해서 새 토큰을 받아야만 정보가 갱신된다
    - JWT의 즉시 무효화 문제가 생기는 이유와 이를 해결하기 위한 방법은 무엇인가요? 🍠
        
        **1️⃣ 이유**
        
        즉시 무효화 문제가 생기는 이유는 **Stateless(무상태)** 특성 때문에 발생한다
        
        1. **토큰의 독립성** : 토큰 자체가 유효 기간 (`exp`) 정보를 들고 있다. 서버가 토큰을 무효라고 하더라도 토큰 안에 유효 시간이 지나지 않으면 서버는 그 토큰을 통과 시켜준다
        2. **서버의 무관심** : 서버는 토큰을 발급한 순간부터 토큰을 추적하지 않는다. 토큰이 들어오면 서명 (signature)가 유효한지 확인한다
            
            ⇒ 유저가 로그아웃을 하거나 관리자가 특정 유저를 탈퇴 시켜도 이미 발급된 토큰은 유효 기간 내에 자유럽게 API를 호출할 수 있게된다
            
        
        **2️⃣ 해결 방법**
        
        1. **AccessToken의 수명을 짧게 설정** 
            - **방법** : AccessToken의 유효 기간을 매우 짧게 잡는다.
            - **효과** : 토큰이 유출되더라도 짧은 시간만 유효하므로 피해를 최소화 할 수 있다.
            - 유저가 짧은 시간마다 로그인 할 수 없으므로 RefreshToken과 함께 사용한다
        2. **RefreshToken과 Token Rotation 활용** 
            - **방법** : 짧은 수명의 AccessToken으로 통신하다가 긴 수명의 RefreshToken을 제출하여 새로운 토큰을 받는다.
            - **효과** : 로그아웃 시 서버 DB에 RefreshToken만 삭제하면 AccessToken이 만료되는 즉시 유저는 더 이상 토큰을 갱신할 수 없기에 자연스레 차단된다
        3. **Blacklist 운영**
            - **방법** : 로그아웃된 토큰이나 유출된 토큰의 ID를 서버 저장소에 등록해둔다. 서버는 토큰의 서명이 맞더라도 토큰이 블랙리스트에 있는지 한 번 더 확인한다.
            - **효과** : 즉시 무효화가 가능하다. 하지만 매 요청마다 저장소를 조회해야하므로 JWT의 장점인 무상태가 퇴색된다는 단점이있다.
        
        **✅ 권장 방식** : 서버 부하 없이 빠르게 검증하고(JWT 장점), 로그아웃이나 보안 사고 시에는 블랙리스트를 확인해 즉시 차단(세션의 장점)하는 **하이브리드 방식**이 좋음
        
    - 세션과 토큰을 결합한 하이브리드 방식(JWT + 저장소 메타데이터)의 동작 원리를 간단히 설명해주세요. 🍠
        
        **1️⃣ 동작원리**
        
        1. **발급 단계 (Login)** : 유저가 로그인하면 서버는 JWT를 통해 JTI(JWT ID)클레임을 활용해 각 토큰에 고유 ID를 부여한다. 서버 측 저장소에 해당 유저의 메타데이터(접속 IP, 기기 정보, RefreshToken 등)을 기록해둔다.
        2. **검증 단계 (Request)** : 유저가 API를 요청 할 때 JWT를 보낸다. 서버는 먼저 서명을 검사하여 위조 여부를 확인한다. 로그아웃 시 이 ID를 블랙리스트에 등록한다.
        3. **조회 단계 (Double Check)** : 요청이 들어와서 토큰을 검증할 때, 블랙리스트에 들어있는지 확인하고 없는 경우 서명을 검증한다.
        
        → JWT의 확장성과 세션의 제어권을 모두 이용한다.
        
        | 구분 | 세션 | JWT 토큰 |
        | --- | --- | --- |
        | 상태 저장 위치 | 서버 | 클라이언트 |
        | 서버 부담 | 높음 (메모리/DB 필요) | 낮음 (서명 검증만) |
        | 즉시 무효화 | 가능 | 어려움 |
        | 확장성 | 제한적 | 뛰어남 |
        | 네트워크 부담 | 낮음 (ID만 전송) | 높음 (토큰 크기) |
        | 적합한 환경 | 단일 서버, 전통적 웹 | MSA, 분산 시스템 |
        
        어떤 방식이든 인증 정보를 클라이언트와 서버 사이에 주고 받아야한다. 
        
        ⇒ **HTTP 요청**에 포함 시켜 전송해야 한다
        
        **2️⃣ 하이브리드 방식의 장점**
        
        JWT는 한 번 발행하면 서버가 제어 할 수 없다는 단점이 있다.  하이브리드 방식을 통해 해결
        
        1. **즉시 로그아웃 가능** : 유저가 로그아웃하면 서버 저장소에서 해당 토큰의 메타 데이터를 삭제하거나 차단 상태로 바꾼다 → 토큰 수명이 남아있어도 다음 요청부터 거부할 수 있다.
        2. **중복 로그인 방지**
        3. **해킹 대응** : 다른 국가나 IP에서 접속하면 서버가 저장소의 데이터를 바탕으로 해당 토큰을 즉시 무효화 (Invalidate) 시킬 수 있다.
- ✅ JWT 정리
    
    **1️⃣ 기존 로그인 방식**
    
    1. 아이디 / 비밀번호 확인
    2. 서버가 “로그인 상태”를 세션에 저장함
    3. 브라우저는 세션 ID를 쿠키로 저장함
    
    👉 문제
    
    - 서버가 모든 사용자 상태를 저장 해야 함
    - 서버가 많아지면 관리가 어려움
    
    **2️⃣ JWT 방식**
    
    로그인 상태를 서버가 저장하지 않음 → **토큰 하나로 로그인 상태를 증명**함
    
    **3️⃣ 흐름**
    
    **🔷 1단계 : 로그인 요청**
    
    	클라이언트 -> 서버
    		*아이디 + 비밀 번호 전송*
    
    **🔷 2단계 : 서버가 토큰 생성**
    
    	로그인을 성공하면 서버는 **JWT(AccessToken)을 생성**함
    	토큰 안에 **사용자 ID, 만료시간, 서명 (signature) 정보**가 들어있음
    
    **🔷 3단계 : 클라이언트가 토큰 저장**
    
    `localStorage` 또는 `쿠키` 등에 저장함
    
    **🔷 4단계 : 이후 요청**
    
    모든 요청에 토큰을 같이 보냄 → `Authorization : Bearer <AccessToken>` 
    
    **🔷 5단계 : 서버가 토큰 검사**
    
    서버는 **DB조회 또는 세션 확인을 하지 않고** **토큰만 보고 판단**함
    
    즉, 서명(signature)와 만료시간만 확인 → 맞으면 “로그인된 사용자”로 인정
    
    **✅ 정리** 
    
    서버가 상태를 저장하지 않는다
    
    이를 `stateless`(무상태)라고 함
    
    **4️⃣ JWT 구조**
    
    JWT는 **3부분**으로 구성되어 있다
    
    ```
    Header.Payload.Signature
    ```
    
    🔷 **Header** : 어떤 방식으로 서명 했는지
    
    ```
    {
      "alg":"HS256", //서명 방식
      "typ":"JWT" // JWT라는 타입
    }
    ```
    
    🔷 **Payload** : 사용자 정보를 나타내는 **실제 정보**가 들어있음 (id, role, 권한 등)
    
    payload에는 **암호화된게 아니라 인코딩된 정보**이므로 누구나 볼 수 있다 → 즉, 민감한 정보 작성 ❌
    
    ```
    {
      "userNickName": "dongdong",
      "name":"bukyung",
      "exp":1710000000 // 만료 시간
    }
    ```
    
    🔷 **Signature** : 위의 내용이 변조되지 않았는지 확인함
    
    ```
    //구조
    Signature = Header + Payload + SECRET_KEY
    ```
    
    서버만 알고 있는 비밀키로 생성된다.
    
    **5️⃣ 장점**
    
    - 서버가 상태를 저장하지 않으므로 확장성이 좋다
    - DB 조회가 없으므로 빠르다
    
    **6️⃣ 단점**
    
    - 토큰이 탈취되면 위험하다
    - 서버에서 강제 로그아웃이 어렵다
- ✅ 세션(Session) 정리
    
    **서버가 로그인 상태를 기억하는 방식**
    
    **1️⃣ 흐름**
    
    **🔷 1단계 : 로그인**
    
    클라이언트 → 서버 : 아이디와 비밀번호 전송함
    
    **🔷 2단계 : 서버가 세션 생성**
    
    1. 로그인 성공
    2. 세션 ID 생성 (예: `abc123`)
    3. 서버 메모리나 Redis에 저장
    
    ```
    서버 저장소:
    abc123 → userId = 1
    ```
    
    [서버] : 매 요청 마다 인증할 때 서버가 조회함
    
    ```
    세션ID → 서버 조회 → 인증
    ```
    
    **🔷 3단계 : 클라이언트에 쿠키 전달**
    
    서버 → 브라우저 : 브라우저가 자동 저장
    
    클라이언트는 사용자 정보는 없고 세션ID만 가짐
    
    ```
    Set-Cookie: sessionId=abc123
    ```
    
    **🔷4단계 : 이후 요청**
    
    브라우저가 자동으로 쿠키를 포함해서 보냄
    
    ```
    Cookie: sessionId=abc123
    ```
    
    🔷 5단계 : 서버 인증
    
    1. 쿠키에서 sessionID를 꺼냄
    2. 저장소에서 조회함
    
    → 로그인된 사용자로 인정
    
    ```
    abc123 → userId = 1
    ```
    
     **✅ 정리**
    
    서버가 sessionID를 저장하고 그걸 기준으로 사용자를 구분함
    
    **2️⃣ JST vs Session**
    
    | 방식 | 누가 상태를 기억함? |
    | --- | --- |
    | 세션 | 서버 |
    | JWT | 토큰 |
    
    **3️⃣ session에서 중요한 요소 3개**
    
    **🔷 세션 ID**
    
    로그인한 사용자를 식별하는 키
    
    서버가 생성함
    
    **🔷 세션 저장소**
    
    서버 메모리 / Redis
    
    실제 로그인 정보가 저장됨
    
    **🔷 쿠키**
    
    브라우저에 저장됨
    
    세션 ID 전달 역할
    
    ```
    // 흐름 구조도
    
    [브라우저]
      ↓ (쿠키)
    sessionId=abc123
    
    [서버]
      ↓ (조회)
    abc123 → 사용자 정보
    ```
    
    **4️⃣ stateful 인 이유?**
    
    서버에서는 `abc123`는 로그인 된 사용자로 기억하고 있음 →  **stateful(상태있음)** 이라 함
    
    5️⃣ **세션 저장소를 어디에 두는가?**
    
    **🔷 메모리 저장 (단일 서버)**
    
    가장 빠르다는 장점이 있지만 서버 재시작 시 모든 세션이 사라진다
    
    **🔷 Redis와 같은 인메모리 데이터베이스 사용 (분산 환경)**
    
    속도와 영속성을 모두 얻을 수 있다.  분산 환경에서 여러 서버가 세션을 공유 할 수 있고 서버를 재시작해도 세션이 유지된다
    
    **🔷 관계형 데이터 베이스 저장**
    
    가장 안정적이지만 다른 방식에 비해 성능이 떨어진다

    - HTTP는 왜 무상태(Stateless)로 설계되었나요? 🍠
        
        1️⃣ 서버의 부담 감소 (가벼운 서버)
        
        서버가 클라이언트의 이전 상태(로그인 여부, 장바구니 담기 등)를 모두 기억해야한다면 사용자가 많이 증가할때 서버는 상태 정보를 메모리에 들고 있어야 하므로 중요한 데이터 처리 (계산, 응답)을 못하게 된다
        
        따라서 HTTP를 무상태로 만들면 서버는 지금 들어온 요청만 딱 처리하고 바로 잊을 수 있다.즉, 서버 메모리를 아주 효율적으로 쓸 수 있다
        
        2️⃣ 서버 확장성 (Scalability) → 스케일 아웃
        
        - **상태가 있다면(Stateful):** 1번 서버에서 로그인한 유저는 반드시 1번 서버로만 요청을 보내야 한다. 다른 서버는 이 유저가 누구인지 모르기에 특정 서버에만 과부하가 걸려도 옮길 수가 없다
        - **무상태라면(Stateless):** 유저의 요청에 필요한 모든 정보(토큰 등)가 이미 담겨 있기 때문에, **어떤 서버가 요청을 받아도 똑같이 처리**할 수 있다. 서버를 맘대로 늘리고 줄이는 게 훨씬 쉬워진다
        
        3️⃣ 단순성과 독립성
        
        - 각각의 요청은 앞뒤 맥락에 의존하지 않는 독립적인 존재이다. 통신 중에 서버가 잠깐 꺼졌다 켜져도, 클라이언트가 다시 요청만 보내면 아무 일 없었다는 듯이 처리가 가능하다. 서버가 이전의 '기억'을 복구할 필요가 없기 때문이다
        
    - HTTP의 무상태성이 주는 장점과 단점을 각각 정리해주세요. 🍠
        
        ## 1. HTTP 무상태성(Stateless)의 장점
        
        **① 서버의 확장성(Scalability) 극대화**
        서버가 클라이언트의 상태를 저장하지 않기 때문에, 수천 대의 서버를 동시에 운영하는 **분산 환경**에서 매우 유리하다
        
        - 어떤 유저의 요청이 1번 서버로 가든, 100번 서버로 가든 상관없습니다. 요청에 필요한 모든 정보가 이미 담겨 있기 때문에 서버를 자유롭게 늘리고 줄이는 **'스케일 아웃(Scale-out)'**이 가능해집니다.
        
        **② 서버 자원의 효율적 사용**
        서버가 수만 명의 사용자 상태를 일일이 메모리에 유지할 필요가 없습니다.
        
        - 요청이 들어올 때만 응답하고 바로 잊어버리면 되기 때문에 서버 메모리 부하가 획기적으로 줄어들고, 더 많은 요청을 동시에 처리할 수 있는 **가벼운 서버**를 만들 수 있습니다.
        
        **③ 서비스의 독립성과 단순성**
        각 요청이 이전 요청에 의존하지 않는 **독립적인 존재**가 됩니다.
        
        - 만약 통신 중에 서버가 잠시 고장 났다가 복구되어도, 클라이언트가 다시 요청을 보내기만 하면 됩니다. 서버가 과거의 기억을 복구하기 위해 애쓸 필요가 없으므로 시스템 전체의 안정성이 높아집니다.
        
        ---
        
        ## 2. HTTP 무상태성(Stateless)의 단점
        
        **① 데이터 전송량의 증가**
        서버가 나를 기억하지 못하기 때문에, 매 요청마다 증명하는 정보(토큰, 사용자 정보 등)를 전부 다 보내야 합니다.
        
        - 로그인을 유지하기 위해 매번 무거운 토큰을 헤더에 실어 보내야 하므로, 상태를 기억하는 방식(Stateful)에 비해 네트워크 대역폭을 더 많이 소모하게 됩니다.
        
        **② 사용자 경험(UX) 구현의 번거로움**
        웹사이트는 원래 '로그인 유지', '쇼핑몰 장바구니'처럼 **상태가 유지되어야 하는 기능**이 필수적입니다.
        
        - 하지만 HTTP 자체가 이를 지원하지 않기 때문에, 개발자가 직접 **쿠키(Cookie), 세션(Session), 토큰(JWT)** 같은 별도의 기술을 사용하여 억지로 상태를 유지시키는 로직을 추가해야 합니다. 이 과정에서 보안이나 저장 위치에 대한 복잡한 설계가 필요해집니다.
        
        **③ 반복적인 인증 절차**
        서버 입장에서는 매번 들어오는 요청이 "처음 보는 사람"의 요청입니다.
        
        - 따라서 매번 토큰의 서명을 검증하거나 복호화하는 등의 **추가적인 계산 과정**이 필요하게 되어, 아주 미세하지만 서버의 연산 자원을 계속 사용하게 됩니다.
    - 쿠키의 Domain 디렉티브에 대해 정리해주세요. 🍠
        
        ## 1. Domain 디렉티브란?
        
        쿠키의 **Domain 디렉티브**는 브라우저가 생성된 쿠키를 "어떤 서버 주소로 다시 보내줄 것인가?"를 결정하는 범위 설정입니다.
        
        ### **① 설정을 생략했을 때 (기본값)**
        
        - **범위:** 쿠키를 발행한 현재 호스트(도메인)에서만 쿠키가 유효합니다.
        - **특징:** 서브 도메인(Sub-domain)을 포함하지 않습니다.
            - 예: `yolog.co.kr`에서 쿠키를 구우면, `yolog.co.kr`에서는 전송되지만 `api.yolog.co.kr`이나 `blog.yolog.co.kr` 같은 서브 도메인에서는 쿠키가 전송되지 않습니다.
        - **보안:** 범위가 현재 페이지로 한정되므로 상대적으로 더 안전합니다.
        
        ### **② 설정을 명시했을 때**
        
        - **범위:** 지정한 도메인과 **그 도메인의 모든 서브 도메인**까지 쿠키가 유효해집니다.
        - **특징:** 도메인 앞에 점(`.`)을 찍거나 생략하여 명시하면 범위가 확장됩니다.
            - 예: `Domain=yolog.co.kr`로 설정하면, 본진인 `yolog.co.kr`은 물론이고 `api.yolog.co.kr`, `dev.yolog.co.kr` 등 어디서든 해당 쿠키를 사용할 수 있습니다.
        
        ---
        
        ## 2. 왜 중요할까? (사용 시나리오)
        
        **① 서비스 통합 로그인 (SSO)**
        여러 개의 서비스(메일, 카페, 블로그 등)를 운영하는 포털 사이트의 경우, 사용자가 한 번만 로그인해도 모든 서브 도메인에서 로그인 상태가 유지되어야 합니다. 이때 Domain 디렉티브를 상위 도메인으로 설정하여 쿠키를 공유합니다.
        
        **② 보안 관리**
        반대로, 특정 API 서버에서만 사용해야 할 민감한 쿠키가 엉뚱한 서브 도메인 사이트로 전송되면 보안 사고가 날 수 있습니다. 이럴 때는 Domain 설정을 생략하여 범위를 최소화하는 것이 정석입니다.
        
        | **설정 방식** | **쿠키 전송 범위** | **특징** |
        | --- | --- | --- |
        | **미지정 (생략)** | 발행한 도메인 **본인만** | 가장 좁은 범위, 보안상 권장 |
        | **지정 (예: yolog.co.kr)** | 본인 + **모든 서브 도메인** | 여러 서버 간 쿠키 공유 가능 |
    - 쿠키의 Path 디렉티브에 대해 정리해주세요. 🍠
        
        ## 1. Path 디렉티브란?
        
        **Path 디렉티브**는 브라우저가 생성된 쿠키를 서버로 보낼 때, “URL 경로가 어디일 때만 이 쿠키를 실어 보낼 것인가?"를 결정하는 설정입니다.
        
        ### **① 설정을 생략했을 때 (기본값)**
        
        - 쿠키가 생성된 **현재 페이지의 경로**가 기본값으로 설정됩니다.
        
        ### **② 설정의 동작 원리 (전방 일치)**
        
        - 설정된 경로와 **그 하위 경로**에서만 쿠키가 전송됩니다.
        - 예를 들어, `Path=/movie`라고 설정하면 다음과 같이 동작합니다.
            - `/movie` (전송 O)
            - `/movie/popular` (전송 O - 하위 경로 포함)
            - `/about` (전송 X - 경로 불일치)
        
        ---
        
        ## 2. 주요 특징 및 주의사항
        
        - **상위 경로는 접근 불가:** 하위 경로에서 생성된 쿠키는 상위 경로로 전송되지 않습니다. 예를 들어 `/movie/detail`에서 생성된 쿠키는 `/movie` 페이지로 이동할 때 전송되지 않을 수 있습니다.
        - **가장 흔한 설정 (`Path=/`):** 웹사이트 전체에서 로그인을 유지하거나 공통 데이터를 사용해야 하는 경우, 경로를 루트(`/`)로 설정합니다. 이렇게 하면 사이트 내 어떤 페이지를 가든 쿠키가 따라갑니다.
        - **보안 강화:** 특정 API(`예: /api/payment`)에서만 사용해야 하는 보안 토큰의 경우, 경로를 해당 API 주소로 한정하여 다른 페이지에서 쿠키가 노출되는 것을 방지할 수 있습니다.
        
        | **구분** | **내용** |
        | --- | --- |
        | **정의** | 쿠키가 전송될 수 있는 **URL 경로 범위**를 지정하는 설정 |
        | **일치 방식** | 지정된 경로와 **그 하위 경로**까지 모두 포함 (Prefix matching) |
        | **일반적 활용** | 사이트 전체 공유를 위해 보통 `Path=/`로 설정하여 사용함 |
    - 세션 쿠키와, 영속 쿠키의 차이점을 정리해주세요. 🍠
        
        ## 1. 세션 쿠키 (Session Cookie)
        
        - "브라우저가 켜져 있는 동안만 살아있는 임시 쿠키"입니다.
        - **설정 방식:** 쿠키를 생성할 때 만료 날짜(`Expires`)나 유효 기간(`Max-Age`)을 **따로 지정하지 않으면** 자동으로 세션 쿠키가 됩니다.
        - **저장 위치:** 브라우저의 메모리(RAM)에 저장됩니다.
        - **삭제 시점:** 사용자가 브라우저를 **완전히 종료하는 순간** 메모리에서 즉시 삭제됩니다.
        - **주요 용도:** 사이트를 돌아다니는 동안 유지되어야 하는 임시 데이터.
            - 보안이 중요한 은행 사이트 등에서 브라우저를 닫으면 자동으로 로그아웃되게 할 때 사용합니다.
        
        ---
        
        ## 2. 영속 쿠키 (Persistent Cookie)
        
        - "브라우저를 닫아도 지정된 시간까지 살아남는 쿠키"입니다.
        - **설정 방식:** 쿠키 생성 시 `Expires`나 `Max-Age` 속성을 사용하여 **미래의 특정 시점이나 초 단위 기간을 명시**해야 합니다.
        - **저장 위치:** 브라우저 종료와 상관없이 사용자의 하드디스크(로컬 파일)에 저장됩니다.
        - **삭제 시점:** 브라우저를 닫아도 사라지지 않으며, 설정한 **만료 시간이 지나야만** 삭제됩니다. (수동으로 삭제하지 않는 한 유지됩니다.)
        - **주요 용도:**
            - "로그인 상태 유지" 체크박스를 눌렀을 때.
            - "오늘 하루 이 창 보지 않기" 설정.
            - 사용자의 설정값(언어, 다크모드 등) 저장
        
        | **구분** | **세션 쿠키 (Session)** | **영속 쿠키 (Persistent)** |
        | --- | --- | --- |
        | **만료 설정** | 별도 설정 없음 | `Expires` 또는 `Max-Age` 지정 |
        | **저장 위치** | 브라우저 메모리 (RAM) | 클라이언트 하드디스크 (Disk) |
        | **수명** | 브라우저 종료 시 삭제 | 만료 날짜 도래 시 삭제 |
        | **비유** | 일회용 출입증 | 상시 출입 카드 |
    - 쿠키의 보안 속성(HttpOnly, Secure, SameSite)은 각각 어떤 공격을 방어하나요? 🍠
        
        ## 1. HttpOnly: XSS 공격 방어
        
        **"자바스크립트로는 쿠키를 절대 건드리지 마!"**
        
        - **동작 원리:** 브라우저에서 `document.cookie` 같은 자바스크립트 코드로 쿠키에 접근하는 것을 원천 봉쇄합니다. 쿠키는 오직 HTTP 통신(서버로 전송될 때)에만 사용됩니다.
        - **방어하는 공격:** **XSS (Cross-Site Scripting)**
            - 해커가 게시판 등에 악성 스크립트를 심어, 유저가 그 글을 읽을 때 유저의 세션 쿠키를 탈취해가는 공격을 막아줍니다. 스크립트가 실행되더라도 `HttpOnly`가 설정된 쿠키는 읽을 수 없기 때문입니다.
        
        ---
        
        ## 2. Secure: 통신 가로채기 방어
        
        **"암호화된 통신(HTTPS)에서만 쿠키를 보내!"**
        
        - **동작 원리:** HTTPS 프로토콜을 사용할 때만 브라우저가 쿠키를 서버로 전송하도록 제한합니다. 일반적인 HTTP 연결에서는 쿠키를 보내지 않습니다.
        - **방어하는 공격:** **Sniffing (네트워크 도청)**
            - 공용 와이파이 같은 보안이 취약한 네트워크에서 해커가 통신 데이터를 가로채더라도, 정보가 암호화되지 않은 HTTP 상태에서는 쿠키 자체가 전송되지 않으므로 쿠키 노출을 방지할 수 있습니다.
        
        ---
        
        ## 3. SameSite: CSRF 공격 방어
        
        **"엉뚱한 사이트에서 보낸 요청에는 쿠키를 싣지 마!"**
        
        - **동작 원리:** 쿠키를 발행한 도메인과 현재 요청을 보내는 도메인이 같은지 확인합니다. (Strict, Lax, None 세 가지 모드가 있습니다.)
        - **방어하는 공격:** **CSRF (Cross-Site Request Forgery, 사이트 간 요청 위조)**
            - 해커가 만든 가짜 사이트에서 유저가 버튼을 눌렀을 때, 유저도 모르게 기존 로그인된 사이트(예: 은행)로 "송금해라"라는 요청을 보내는 공격입니다. `SameSite` 속성이 있으면 외부 사이트에서 오는 요청에는 쿠키를 포함하지 않으므로 서버가 해당 요청을 거부하게 됩니다.
            
            | **속성** | **방어하는 핵심 공격** | **한 줄 정의** |
            | --- | --- | --- |
            | **HttpOnly** | **XSS** (스크립트 탈취) | 자바스크립트 접근 차단 |
            | **Secure** | **Sniffing** (네트워크 도청) | HTTPS 연결에서만 전송 |
            | **SameSite** | **CSRF** (요청 위조) | 타 사이트에서의 쿠키 전송 제한 |
    - 쿠키의 한계점(용량, 보안, 네트워크, 도메인 제약)을 정리해주세요. 🍠
        
        ## 1. 용량 및 개수 제한 (Storage Limit)
        
        **"가벼운 정보만 담을 수 있어요"**
        
        - **용량 제한:** 하나의 쿠키는 약 **4KB**를 넘을 수 없습니다. 이는 복잡한 사용자 설정이나 대량의 데이터를 담기에는 턱없이 부족한 크기입니다.
        - **개수 제한:** 브라우저마다 다르지만, 하나의 도메인당 약 **20~50개**, 전체 브라우저에 약 **300개** 정도로 쿠키의 개수가 제한됩니다.
        - **결과:** 많은 양의 데이터를 로컬에 저장해야 할 때는 쿠키 대신 Web Storage(LocalStorage 등)를 사용해야 합니다.
        
        ---
        
        ## 2. 보안 취약성 (Security Vulnerabilities)
        
        **"클라이언트에 노출되어 있어 위험해요"**
        
        - **노출 위험:** 쿠키는 사용자의 기기에 파일 형태로 저장되거나 브라우저 메모리에 남습니다. 따라서 PC를 분실하거나 공용 PC를 사용할 경우 타인에게 노출될 위험이 큽니다.
        - **공격 표적:** 앞서 공부했듯이 **XSS**(스크립트 탈취)나 **CSRF**(요청 위조) 공격의 타겟이 되기 쉽습니다. 비록 보안 속성(`HttpOnly`, `SameSite`)이 생겼지만, 근본적으로 클라이언트에 정보가 있다는 점은 보안상 취약점입니다.
        
        ---
        
        ## 3. 네트워크 성능 저하 (Network Overhead)
        
        **"불필요한 데이터까지 매번 같이 전송돼요"**
        
        - **자동 전송:** 쿠키의 가장 큰 특징이자 단점은, 한 번 설정되면 해당 도메인으로 보내는 **모든 HTTP 요청에 자동으로 포함**된다는 것입니다.
        - **오버헤드:** 이미지가 많은 페이지에서 이미지 하나하나를 불러올 때마다 로그인을 위한 쿠키가 계속 실려 나간다면, 실제 데이터보다 쿠키 값이 더 커지는 배보다 배꼽이 더 큰 상황이 생겨 **네트워크 속도 저하**를 일으킵니다.
        
        ---
        
        ## 4. 도메인 및 경로 제약 (Domain & Path Constraints)
        
        **"울타리를 넘기 어려워요"**
        
        - **단일 도메인 원칙:** 쿠키는 기본적으로 이를 생성한 도메인 내에서만 유효합니다. 만약 내가 운영하는 사이트가 `a.com`과 `b.com` 두 개일 때, 사용자가 `a.com`에서 로그인했더라도 `b.com`으로 이동하면 다시 로그인해야 합니다.
        - **공유의 복잡성:** 서브 도메인 간 공유는 `Domain` 설정으로 가능하지만, 완전히 다른 도메인(Cross-Domain) 간의 데이터 공유는 매우 까다롭고 보안 정책상 제한이 많습니다
        
        | **한계점** | **주요 내용** |
        | --- | --- |
        | **용량** | 4KB 이내의 소량 데이터만 저장 가능 |
        | **보안** | 로컬 저장 방식으로 인한 탈취 및 위변조 위험 존재 |
        | **네트워크** | 매 요청마다 자동 전송되어 불필요한 트래픽 유발 |
        | **제약** | 생성된 도메인 밖에서는 기본적으로 사용 불가 |
    - 쿠키만으로 상태 관리를 해결할 수 없는 이유는 무엇인가요? 🍠
        
        ## 1. 보안의 취약성 (Security Risks)
        
        가장 큰 이유는 **보안**입니다. 쿠키는 사용자 브라우저에 평문으로 저장되거나 쉽게 노출될 수 있는 위치에 있습니다.
        
        - **데이터 변조:** 사용자가 직접 브라우저 설정에서 쿠키 값을 수정할 수 있습니다. 예를 들어, 쿠키에 `is_admin=false`라고 적혀 있다면 유저가 이를 `true`로 바꿔서 관리자 권한을 가로챌 수 있습니다.
        - **공격 노출:** 앞서 공부한 **XSS**(스크립트 탈취)나 **CSRF**(요청 위조) 공격에 매우 취약하여, 민감한 개인정보나 결제 상태 등을 쿠키에만 의존해 저장하기에는 매우 위험합니다.
        
        ---
        
        ## 2. 저장 용량의 한계 (Storage Limit)
        
        현대의 웹 서비스는 사용자별 다크모드 설정, 장바구니 리스트, 최근 본 상품 등 관리해야 할 상태가 아주 많습니다.
        
        - **4KB의 벽:** 하나의 쿠키는 최대 **4KB**까지만 저장할 수 있습니다. 조금만 복잡한 데이터를 담으려고 해도 용량이 부족해져서 데이터가 잘리거나 저장이 되지 않습니다.
        - **개수 제한:** 도메인당 가질 수 있는 쿠키 개수도 제한되어 있어, 대규모 서비스의 방대한 상태를 모두 담기에는 그릇이 너무 작습니다.
        
        ---
        
        ## 3. 네트워크 성능 저하 (Performance Overhead)
        
        쿠키의 특징 중 하나는 "해당 도메인의 모든 요청에 자동으로 포함된다"는 것입니다.
        
        - **불필요한 트래픽:** 로그인을 위해 만들어둔 쿠키가 영화 포스터 이미지, CSS 파일, 아이콘 등을 불러오는 모든 요청에 매번 따라붙습니다.
        - **배보다 배꼽이 큰 상황:** 요청 본문 데이터는 아주 작은데, 쿠키 값이 담긴 헤더가 더 커지는 현상이 발생하여 네트워크 대역폭을 낭비하고 페이지 로딩 속도를 늦추는 원인이 됩니다.
        
        ---
        
        ## 4. 단순한 데이터 구조
        
        쿠키는 오직 **'문자열(String)'** 형태의 데이터만 저장할 수 있습니다.
        
        - 현대의 상태 관리에서는 객체(Object)나 배열(Array) 형태의 복잡한 데이터를 다루어야 할 때가 많습니다.
        - 쿠키에 이를 저장하려면 매번 직렬화(JSON.stringify)와 역직렬화(JSON.parse) 과정을 거쳐야 하며, 이는 개발 생산성을 떨어뜨리고 에러 발생 가능성을 높입니다.
        
        ✅ 요약 : 쿠키는 보안이 취약하고 용량이 매우 작으며(4KB), 모든 요청에 자동으로 포함되어 네트워크 성능을 저하시키는 특성이 있습니다. 따라서 보안이 중요한 인증 정보는 서버(세션)나 암호화된 토큰(JWT)으로, 큰 용량의 클라이언트 상태는 로컬 스토리지로 분산하여 관리하는 것이 현대 웹의 표준입니다

    - XSS 공격은 무엇인가요?
        
        **XSS**는 해커가 보안이 취약한 웹사이트에 악성 스크립트(주로 자바스크립트)를 삽입하여, 해당 사이트를 방문하는 일반 사용자의 브라우저에서 그 스크립트가 실행되게 만드는 공격입니다.
        
        - **핵심 원리:** 웹사이트가 사용자로부터 입력받은 값을 적절한 검사 없이 그대로 화면에 출력할 때 발생합니다.
        - **주요 피해:** * **세션 탈취:** 사용자의 쿠키(세션 ID)를 가로채서 로그인 권한을 뺏음.
            - **개인정보 유출:** 사용자가 입력하는 아이디, 비밀번호 등을 가로챔.
            - **악성 행위:** 유저의 의사와 상관없이 게시글을 작성하거나 결제 요청을 보냄.
    - XSS 방어 전략에 대해 정리해주세요.
        
        ## 2. XSS 방어 전략
        
        XSS를 완벽히 막기 위해서는 입력값 검증부터 브라우저 보안 설정까지 여러 겹의 방어막이 필요합니다.
        
        ### ① 입출력 값 필터링 및 이스케이프 (Filtering & Escaping)
        
        - **이스케이프(Escaping):** `<`를 `&lt;`, `>`를 `&gt;`와 같이 HTML 엔티티로 변환하는 것입니다. 이렇게 하면 브라우저는 이를 '코드'가 아닌 '일반 문자열'로 인식하여 실행하지 않습니다.
        - **필터링(Filtering):** `<script>`, `onerror`, `onload`와 같은 위험한 태그나 이벤트 핸들러를 서버 단에서 미리 제거하거나 수정하는 방식입니다.
        
        ### ② HttpOnly 쿠키 설정
        
        - 가장 확실한 세션 탈취 방지책입니다. 쿠키를 생성할 때 **`HttpOnly`** 속성을 부여하면, 자바스크립트(`document.cookie`)로는 해당 쿠키에 접근할 수 없게 됩니다. 해커의 악성 스크립트가 실행되더라도 쿠키를 빼갈 수 없습니다.
        
        ### ③ CSP (Content Security Policy) 설정
        
        - 브라우저에게 "이 사이트에서는 우리가 신뢰하는 출처의 스크립트만 실행해!"라고 명령하는 보안 정책입니다.
        - HTTP 헤더에 CSP를 설정하면, 해커가 외부에서 강제로 삽입한 인라인 스크립트나 알 수 없는 출처의 파일 실행을 브라우저가 원천 차단합니다.
        
        ### ④ 라이브러리 및 프레임워크 활용
        
        - React, Vue 같은 현대적인 프레임워크는 기본적으로 변수를 출력할 때 자동으로 이스케이프 처리를 해줍니다. (예: `{content}`로 출력하면 태그가 문자열로 변함)
        - 위험한 HTML을 직접 렌더링해야 한다면 `DOMPurify` 같은 검증된 라이브러리를 사용하여 악성 코드를 소독(Sanitize)한 뒤 사용해야 합니다.

    - 세션 하이재킹은 무엇인가요? 🍠
        
        **세션 하이재킹**은 단어 뜻 그대로 사용자와 서버 사이의 '세션(연결 상태)'을 '하이재킹(납치)'하는 공격 수법입니다.
        
        - **핵심 원리:** 사용자가 로그인을 하면 서버는 그 사용자를 식별하기 위해 **세션 ID**라는 신분증을 발급해 줍니다. 해커가 어떤 수단을 써서든 이 **세션 ID를 가로채면**, 서버는 해커를 정상적인 사용자로 착각하게 됩니다.
        - **공격의 특징:** 아이디나 비밀번호를 몰라도 됩니다. 이미 로그인이 완료된 상태의 '세션 ID'만 있으면 되기 때문에, 해커는 별도의 인증 절차 없이 바로 사용자의 계정 권한을 행사할 수 있습니다.
        - **비유:** 호텔 프런트에서 발급받은 '객실 카드키'를 누군가 몰래 훔쳐서 방에 들어가는 것과 같습니다. 호텔 직원(서버)은 카드키를 가진 사람을 방 주인으로 믿고 문을 열어주는 것이죠.
    - XSS 공격을 차단하는 방법은 무엇인가요? 🍠
        
        ### **1. HttpOnly 쿠키 속성 사용**
        
        세션 하이재킹의 가장 흔한 경로인 `document.cookie` 접근을 차단하는 방법입니다.
        
        - **원리:** 서버가 세션 ID를 쿠키에 담아 보낼 때 **`HttpOnly`** 플래그를 설정합니다.
        - **효과:** 브라우저에 악성 스크립트가 침투하더라도, 자바스크립트 명령으로는 쿠키 데이터를 읽을 수 없게 됩니다. 해커가 신분증(세션 ID)을 훔쳐가는 가장 쉬운 길을 봉쇄하는 것입니다.
        
        ### **2. 입력값 검증 및 출력값 이스케이프 (Escaping)**
        
        악성 스크립트 자체가 웹 페이지에 삽입되지 않도록 하는 기본 방어입니다.
        
        - **원리:** 사용자가 입력한 모든 데이터를 '코드'가 아닌 '단순 문자열'로 처리합니다.
        - **방법:** `<`는 `&lt;`, `>`는 `&gt;`로 변환하여 브라우저가 스크립트 태그(`<script>`)를 실행하지 못하고 텍스트로만 화면에 그리게 만듭니다.
        
        ### **3. CSP (Content Security Policy) 도입**
        
        브라우저 차원에서 실행 가능한 자원의 출처를 제한하는 정책입니다.
        
        - **원리:** HTTP 헤더에 "우리 사이트가 허용한 도메인에서 온 스크립트만 실행해!"라는 지침을 내립니다.
        - **효과:** 해커가 외부 서버에서 몰래 불러오려는 악성 스크립트 파일이나, 인라인으로 직접 삽입한 코드의 실행을 브라우저가 스스로 거부합니다.
        
        ### **4. 세션 만료 및 재발급 주기 단축**
        
        공격자가 세션 ID를 가로챘더라도 사용할 수 있는 시간을 최소화하는 전략입니다.
        
        - **방법:** 세션 유지 시간(Timeout)을 짧게 설정하고, 로그인 시마다 혹은 특정 주기마다 세션 ID를 새로 발급(Session Regeneration)합니다.
        - **효과:** 해커가 가로챈 ID가 금방 폐기되므로 피해 범위를 줄일 수 있습니다.
        
        ✅ 요약 : XSS를 통한 세션 하이재킹을 막기 위해 가장 필수적인 방법은 자바스크립트의 쿠키 접근을 차단하는 HttpOnly 속성을 적용하는 것입니다. 이에 더해 입력된 특수문자를 안전하게 변환하는 이스케이프 처리와, 신뢰할 수 있는 스크립트만 실행하도록 제한하는 CSP 설정을 병행하여 다중 방어 체계를 구축해야 합니다
        
    - 쿠키의 다양한 옵션들에 대해서 정리해주세요. 🍠
        
        ## 쿠키의 핵심 옵션(Directives) 정리
        
        쿠키를 설정할 때 뒤에 붙는 옵션들은 쿠키의 **수명, 범위, 그리고 보안**을 결정하는 아주 중요한 설정값들입니다.
        
        ### 1. 수명 관련 옵션 (Life Cycle)
        
        - **Expires:** 쿠키가 삭제될 **정확한 날짜와 시간**을 지정합니다. (예: `Expires=Wed, 21 Oct 2026 07:28:00 GMT`)
        - **Max-Age:** 현재 시간을 기준으로 **몇 초 동안** 쿠키를 유지할지 결정합니다. (예: `Max-Age=3600`은 1시간)
            - **참고:** 이 두 옵션이 없으면 브라우저 종료 시 사라지는 **세션 쿠키**가 되고, 설정하면 하드에 저장되는 **영속 쿠키**가 됩니다.
        
        ### 2. 범위 관련 옵션 (Scope)
        
        - **Domain:** 쿠키가 전송될 **도메인 범위**를 정합니다. 설정을 생략하면 현재 도메인에서만 사용되고, 명시하면 모든 서브 도메인(api.xxx, dev.xxx 등)까지 전송됩니다.
        - **Path:** 특정 **URL 경로**에서만 쿠키를 보내도록 제한합니다. 보통 사이트 전체에서 쓰기 위해 `Path=/`로 설정하는 경우가 많습니다.
        
        ### 3. 보안 관련 옵션 (Security) - 매우 중요!
        
        - **HttpOnly:** 자바스크립트(`document.cookie`)로 쿠키에 접근하는 것을 막습니다. **XSS 공격을 통한 세션 탈취를 방어**하는 일등 공신입니다.
        - **Secure:** **HTTPS(암호화 통신)** 연결일 때만 쿠키를 전송합니다. 일반 HTTP 환경에서 쿠키가 도청(Sniffing)당하는 것을 방지합니다.
        - **SameSite:** 타 사이트에서 우리 사이트로 보내는 요청에 쿠키를 실을지 결정합니다. **CSRF(사이트 간 요청 위조) 공격을 방어**합니다.
            - `Strict`: 우리 사이트 내부 이동일 때만 전송.
            - `Lax`: 기본값. 링크를 타고 들어올 때는 허용하지만, 타 사이트의 위험한 요청은 차단.
            - `None`: 모든 곳에서 전송 허용 (`Secure` 속성이 필수)
        
        | **옵션 구분** | **종류** | **역할** |
        | --- | --- | --- |
        | **수명** | `Expires`, `Max-Age` | 쿠키가 언제 삭제될지 결정 (세션 vs 영속) |
        | **범위** | `Domain`, `Path` | 어떤 도메인과 경로로 쿠키를 보낼지 결정 |
        | **보안** | `HttpOnly`, `Secure`, `SameSite` | 해킹 공격(XSS, Sniffing, CSRF)으로부터 쿠키 보호 |

    - CSRF는 무엇인가요? 🍠
        
        ### **CSRF(Cross-Site Request Forgery)란?**
        
        **CSRF**는 우리말로 '사이트 간 요청 위조'라고 부릅니다. 사용자가 자신의 의지와는 무관하게, 공격자가 의도한 행위(수정, 삭제, 등록 등)를 특정 웹사이트에 요청하게 만드는 공격 수법입니다.
        
        - **핵심 원리:** 사용자가 특정 사이트(예: 은행)에 이미 로그인되어 있는 상태(쿠키가 브라우저에 남아있는 상태)를 이용합니다.
        - **공격 과정:**
            1. 사용자가 로그인된 상태로 해커가 만든 낚시 사이트나 게시글을 클릭합니다.
            2. 그 페이지에 숨겨진 악성 코드(이미지 태그, 폼 태그 등)가 실행됩니다.
            3. 브라우저는 "어? 이 사이트로 요청을 보내네? 로그인 쿠키가 있으니까 같이 보내야지!" 하고 **사용자의 인증 정보를 실어서** 서버에 요청을 보냅니다.
            4. 서버는 "정상적인 유저가 보낸 요청이 맞구나!"라고 착각하고 송금이나 비밀번호 변경 같은 명령을 수행합니다.
        
        ---
        
        ### **XSS와 무엇이 다른가요?**
        
        - **XSS:** 사용자의 브라우저에서 스크립트를 실행해 **정보(쿠키 등)를 탈취**하는 것이 목적입니다.
        - **CSRF:** 사용자의 정보를 빼가는 것보다, 사용자의 권한을 이용해 **서버에 가짜 명령을 내리는 것**이 목적입니다. 해커는 유저의 쿠키 내용을 몰라도 공격에 성공할 수 있습니다
        
        ✅ 요약 : CSRF는 사용자가 로그인한 상태로 해커가 심어놓은 링크나 페이지를 방문했을 때, 브라우저가 자동으로 쿠키를 포함해 요청을 보내는 특성을 악용하는 공격입니다. 이를 통해 사용자의 의사와 상관없이 비밀번호 변경, 자금 이체 등의 악의적인 요청을 서버에 전송하여 실행하게 만듭니다
        
    - CSRF 방어 전략에 대해 정리해주세요. 🍠
        
        ## 1. CSRF 토큰(Token) 사용 (가장 보편적인 방법)
        
        사용자의 세션에 임의의 난수 값을 저장하고, 모든 요청(POST, PUT 등) 시 이 값을 함께 보내도록 하는 방법입니다.
        
        - **동작 원리:** 서버는 페이지를 렌더링할 때 숨겨진 필드(`<input type="hidden">`) 등에 **CSRF 토큰**을 심어서 보냅니다. 사용자가 폼을 제출하면 서버는 세션에 저장된 토큰 값과 요청에 담긴 토큰 값이 일치하는지 확인합니다.
        - **방어 원리:** 해커는 사용자의 쿠키는 이용할 수 있지만, 서버가 발행한 이 **일회성 토큰 값**은 알 수 없기 때문에 위조된 요청을 보낼 수 없습니다.
        
        ---
        
        ## 2. SameSite 쿠키 속성 설정
        
        브라우저 차원에서 타 사이트 간의 쿠키 전송을 제어하는 설정입니다.
        
        - **Strict:** 다른 사이트에서 우리 사이트로 가는 모든 요청에 쿠키를 포함하지 않습니다. (가장 안전하지만 사용자 편의성이 낮아질 수 있음)
        - **Lax:** 기본값으로, 위험한 요청(POST 등)에는 쿠키를 보내지 않지만, 단순 링크 클릭(GET)을 통한 이동 시에는 쿠키를 허용합니다.
        - **방어 원리:** 해커의 낚시 사이트에서 우리 서버로 요청을 보낼 때 **쿠키 자체를 실어 보내지 못하게** 차단하여 인증을 실패하게 만듭니다.
        
        ---
        
        ## 3. Double Submit Cookie (더블 서밋 쿠키)
        
        세션을 사용할 수 없는 무상태(Stateless) 환경에서 주로 사용하는 방식입니다.
        
        - **동작 원리:** 쿠키에 랜덤 값을 저장하고, 동시에 요청 파라미터(또는 헤더)에도 같은 값을 실어 보냅니다.
        - **방어 원리:** 서버는 두 값이 일치하는지만 확인합니다. 해커는 스크립트로 타 도메인의 쿠키를 읽을 수 없으므로(Same-Origin Policy), 요청 파라미터에 똑같은 값을 채워 넣을 수 없습니다.
        
        ---
        
        ## 4. Referer 및 Origin 헤더 체크
        
        요청이 어디서 시작되었는지 확인하는 방법입니다.
        
        - **동작 원리:** HTTP 요청 헤더에 포함된 `Referer`(이전 페이지 주소)나 `Origin`(요청을 보낸 도메인) 값을 확인합니다.
        - **방어 원리:** 요청이 우리 서비스의 도메인이 아닌 엉뚱한 사이트(해커의 사이트)에서 출발했다면 서버가 해당 요청을 즉시 거부합니다.
    - CSRF 토큰의 장점과 단점에 대해 정리해주세요. 🍠
        
        ## 1. CSRF 토큰의 장점 (Pros)
        
        - **가장 확실한 보안성:** 쿠키에만 의존하는 방식과 달리, 서버가 매번 발행하는 **고유한 난수 값**을 확인합니다. 해커는 사용자의 쿠키는 이용할 수 있어도, 사용자의 HTML 내부에 숨겨진 이 토큰 값까지는 알 수 없기 때문에 공격을 원천 봉쇄할 수 있습니다.
        - **세션과의 강력한 결합:** 토큰이 사용자의 세션과 연동되어 있어, 해당 세션이 유효한 동안에만 사용할 수 있습니다. 이는 요청의 **출처(Origin)를 증명**하는 가장 강력한 수단이 됩니다.
        - **표준화된 방어 기제:** Spring Security나 Django 같은 현대적인 백엔드 프레임워크에서 기본적으로 지원하는 방식이라, 안정성이 검증되어 있고 도입이 비교적 쉽습니다.
        
        ---
        
        ## 2. CSRF 토큰의 단점 (Cons)
        
        - **서버 자원 사용 (Stateful):** 서버는 발행한 토큰 값을 사용자의 세션에 저장하고 있어야 합니다. 사용자가 많아질수록 서버의 **메모리 부하**가 늘어날 수 있습니다.
        - **무상태(Stateless) 환경에서의 어려움:** REST API나 JWT를 사용하는 '무상태' 환경에서는 서버가 세션을 유지하지 않으므로, 이 토큰 방식을 그대로 적용하기가 까다롭습니다. (이런 경우 앞서 언급한 'Double Submit Cookie' 방식을 대안으로 씁니다.)
        - **복잡한 구현 로직:** 단순한 API 호출이 아니라, 모든 폼 제출이나 상태 변경 요청마다 토큰을 포함해야 하므로 프론트엔드와 백엔드 양쪽에서 **추가적인 작업**이 필요합니다. 특히 비동기 요청(Axios, Fetch) 시 헤더에 토큰을 직접 실어주는 처리를 잊지 말아야 합니다.
        - **사용자 편의성 저하:** 만약 토큰 만료 시간이 짧다면, 사용자가 폼을 작성하는 도중에 토큰이 만료되어 작업 내용이 날아가거나 에러가 발생할 수 있습니다.
    - CAPTCHA는 무엇인가요? 🍠
        
        ### **1. CAPTCHA란 무엇인가요?**
        
        **CAPTCHA**는 "Completely Automated Public Turing test to tell Computers and Humans Apart"의 약자로, 해석하면 "컴퓨터와 인간을 판별하기 위한 완전 자동화된 공용 튜링 테스트"입니다.
        
        - **핵심 원리:** 사람은 쉽게 풀 수 있지만, 컴퓨터 프로그램(봇)은 해결하기 어려운 문제를 제시하여 요청자가 '진짜 사람'인지 확인하는 기술입니다.
        - **방어 목적:** * **스팸 방지:** 게시판에 수만 개의 광고 글을 자동으로 올리는 봇 차단.
            - **어뷰징 방지:** 티켓 예매나 한정판 구매 시 매크로 프로그램을 통한 싹쓸이 방지.
            - **무차별 대입 공격(Brute Force) 방지:** 로그인 시 수억 번의 비밀번호를 대입해 보는 해킹 시도 차단.
        
        ---
        
        ### **2. CAPTCHA의 진화 과정**
        
        1. **텍스트 방식:** 찌그러지거나 노이즈가 섞인 글자를 보고 직접 타이핑하는 초기 방식입니다. (현재는 AI의 문자 인식 능력이 좋아져서 보안성이 낮아졌습니다.)
        2. **이미지 선택 방식:** "신호등이 포함된 사진을 모두 고르세요"와 같은 방식입니다. 사람의 인지 능력을 활용합니다.
        3. **reCAPTCHA (v2/v3):** 구글에서 개발한 방식으로, **"로봇이 아닙니다"** 체크박스를 누르거나(v2), 아예 사용자에게 아무것도 묻지 않고 마우스 움직임이나 접속 패턴을 분석해 점수를 매기는 방식(v3)으로 진화했습니다.
        
        ---
        
        ### **3. CSRF와 CAPTCHA의 관계**
        
        CSRF 공격을 막는 전략 중 하나로 CAPTCHA가 쓰이기도 합니다.
        
        - **이유:** CSRF는 사용자가 모르는 사이에 브라우저가 자동으로 요청을 보내는 방식입니다.
        - **효과:** 중요한 요청(예: 이체, 비밀번호 변경) 직전에 CAPTCHA를 띄우면, **사용자의 직접적인 개입** 없이는 요청을 완료할 수 없으므로 CSRF 공격이 성공할 수 없게 됩니다.

    - 쿠키 방식으로 인증 정보를 전송할 때 브라우저가 자동으로 쿠키를 포함하는 조건은 무엇인가요? 🍠
        
        ### **1. 도메인(Domain) 일치**
        
        가장 먼저 "이 쿠키를 구워준 주인이 맞는가?"를 확인합니다.
        
        - 쿠키가 생성될 때 설정된 `Domain` 값과 현재 요청을 보내려는 서버의 주소가 일치해야 합니다.
        - 만약 `naver.com`에서 발행한 쿠키라면, `google.com`에 요청을 보낼 때는 절대로 가져가지 않습니다.
        
        ### **2. 경로(Path) 일치**
        
        주인이 맞다면 그다지 "상세 주소(URL 경로)도 허용된 범위인가?"를 확인합니다.
        
        - 쿠키의 `Path` 옵션에 지정된 경로와 현재 요청하는 URL의 경로가 일치하거나, 그 하위 경로여야 합니다.
        - 예를 들어 `Path=/user`로 설정된 쿠키는 `/user/profile` 요청에는 포함되지만, `/shop` 요청에는 포함되지 않습니다.
        
        ### **3. 보안 및 연결 설정 (Secure & SameSite)**
        
        마지막으로 "안전한 통로인가?"와 "어디서 온 요청인가?"를 확인합니다.
        
        - **Secure 속성:** 이 속성이 있다면 반드시 **HTTPS** 연결일 때만 쿠키를 포함합니다.
        - **SameSite 속성:** 현재 내가 머물고 있는 사이트와 요청을 보내는 사이트가 다를 경우(Cross-site), 이 설정값(`Strict`, `Lax`, `None`)에 따라 쿠키를 보낼지 말지 최종 결정합니다.
    - 크로스 도메인 환경에서 쿠키를 전송하려면 서버와 클라이언트 측에서 각각 어떤 설정이 필요할까요? 🍠
        
        ### **1. 클라이언트 측 설정 (Client-side)**
        
        요청을 보낼 때 "나 이번에 쿠키도 같이 실어 보낼게!"라고 명시해야 합니다.
        
        - **설정:** `withCredentials: true`
        - **방법:** `fetch` API를 사용할 때는 `credentials: 'include'` 옵션을, `axios`를 사용할 때는 `withCredentials: true` 옵션을 추가합니다.
        - **의미:** 이 설정을 하지 않으면 브라우저가 자동으로 쿠키를 차단합니다.
        
        ### **2. 서버 측 설정 (Server-side)**
        
        서버도 "응, 너네 도메인에서 온 쿠키랑 요청 받아줄게!"라고 응답 헤더를 통해 허락해줘야 합니다.
        
        - **Access-Control-Allow-Origin:** 요청을 보낸 정확한 도메인을 명시해야 합니다. (주의: `withCredentials`를 사용할 때는 와일드카드를 사용할 수 없습니다.)
        - **Access-Control-Allow-Credentials: true:** 이 헤더를 `true`로 설정해야만 브라우저가 서버의 응답을 수락하고 쿠키를 주고받을 수 있습니다.
        
        ### **3. 쿠키 자체 설정 (Cookie Attributes)**
        
        크로스 도메인 환경에서는 쿠키의 보안 속성도 까다로워집니다.
        
        - **SameSite=None:** 서로 다른 도메인 간 전송을 허용하기 위해 이 옵션이 반드시 필요합니다.
        - **Secure:** `SameSite=None`을 설정하려면 반드시 `Secure` 속성(HTTPS 연결)도 함께 설정되어야만 브라우저가 인정해 줍니다.
        
        | **주체** | **필요 설정** | **비고** |
        | --- | --- | --- |
        | **클라이언트** | `withCredentials: true` | 요청 시 쿠키 포함 선언 |
        | **서버** | `Allow-Credentials: true` | 응답 시 쿠키 수용 선언 |
        | **쿠키** | `SameSite=None; Secure` | 크로스 도메인 전송 허용 속성 |
    - 쿠키 기반 인증에서 CSRF 공격이 발생할 수 있는 원리는 무엇인가요? 🍠
        
        ### **1. 브라우저의 자동 메커니즘 (자동 배달)**
        
        브라우저는 특정 도메인에 요청을 보낼 때, 해당 도메인 앞으로 구워진 쿠키가 있다면 **사용자가 시키지 않아도 자동으로 요청 헤더에 실어서** 보냅니다. 브라우저 입장에서는 "주인님 신분증(쿠키) 여기 있으니까 같이 가져가야지!" 하고 친절을 베푸는 것이죠.
        
        ### **2. 요청의 출처를 묻지 않는 서버 (무조건 신뢰)**
        
        서버는 요청이 들어오면 그 안에 들어있는 **쿠키(세션 ID 등)만 확인**합니다.
        
        - 서버 입장: "오, 정상적인 세션 ID가 들어있네? 로그인한 부경님이 보낸 요청이 맞구나!"
        - 현실: 사실 그 요청은 부경님이 직접 누른 게 아니라, 해커가 만든 낚시 페이지가 부경님의 브라우저를 이용해 **몰래 보낸 요청**일 수 있습니다.
        
        ### **3. 공격의 흐름**
        
        1. 사용자가 대상 사이트(A)에 로그인하여 브라우저에 **인증 쿠키**가 남음.
        2. 사용자가 로그아웃하지 않은 상태로 해커가 만든 악성 사이트(B)에 접속.
        3. 사이트(B)에는 사이트(A)로 요청을 보내는 코드(예: `<img src="A사이트/송금">`)가 숨겨져 있음.
        4. 사용자의 브라우저는 사이트(A)로 요청을 보낼 때, **A의 쿠키를 자동으로 포함**시킴.
        5. 사이트(A) 서버는 쿠키만 보고 **정상 요청으로 판단하여 실행**.
        
        ✅ 요약 : CSRF 공격이 발생하는 이유는 브라우저가 특정 도메인으로 요청을 보낼 때 해당 도메인의 쿠키를 자동으로 포함하는 특성 때문입니다. 공격자는 이 점을 악용하여, 사용자가 로그인된 상태를 이용해 사용자의 의사와 무관한 요청을 서버에 전송하며, 서버는 요청의 출처를 확인하지 않고 쿠키 속 인증 정보만 보고 이를 처리하기 때문에 발생합니다
        
    - 헤더 방식 인증의 주요 장점(예: CSRF 방어, 선택적 전송, CORS 단순화)을 정리해주세요. 🍠
        
        ## 1. CSRF 공격으로부터 안전함
        
        헤더 방식인 `Authorization: Bearer <Token>`은 CSRF 공격을 원천 봉쇄할 수 있습니다.
        
        - **이유:** CSRF는 브라우저가 쿠키를 **자동으로** 실어 보내는 특성을 이용합니다. 하지만 헤더 방식은 자바스크립트 코드로 직접 헤더에 값을 넣어줘야만 전송됩니다.
        - **방어 원리:** 해커가 만든 낚시 사이트에서는 사용자의 브라우저를 이용해 요청을 보낼 수는 있어도, 사용자의 토큰 값을 가져와 헤더에 **수동으로** 넣어줄 방법이 없기 때문에 공격이 실패하게 됩니다.
        
        ## 2. 선택적 전송 (필요한 곳에만 전송)
        
        쿠키는 해당 도메인의 모든 요청(이미지, CSS 등)에 무조건 따라붙어 네트워크 낭비를 초래하지만, 헤더 방식은 매우 경제적입니다.
        
        - **원리:** 인증이 꼭 필요한 API 호출에만 개발자가 선택적으로 토큰을 포함할 수 있습니다.
        - **장점:** 불필요한 트래픽(오버헤드)을 줄일 수 있어 전체적인 네트워크 성능 향상에 도움이 됩니다.
        
        ## 3. CORS 및 크로스 도메인 관리의 단순화
        
        서로 다른 도메인 간의 통신에서 쿠키를 쓰려면 `withCredentials`나 `SameSite=None` 등 설정이 복잡하지만, 헤더 방식은 상대적으로 깔끔합니다.
        
        - **원리:** 토큰 정보 자체가 요청 헤더에 텍스트로 포함되기 때문에, 서버에서 해당 헤더를 허용(`Access-Control-Allow-Headers`)해주기만 하면 됩니다.
        - **장점:** 쿠키의 도메인 정책에 얽매이지 않고 여러 서비스(API 서버와 프론트엔드 서버가 다를 때 등) 간의 인증을 더 직관적으로 처리할 수 있습니다.
        
        ## 4. 모바일 앱(Native)과의 호환성
        
        - 쿠키는 웹 브라우저 기반의 기술이라 모바일 네이티브 앱(iOS, Android)에서 관리하기 까다로운 경우가 많습니다.
        - 반면 헤더 방식은 모든 플랫폼에서 **HTTP 표준 헤더**를 사용하므로, 플랫폼에 상관없이 동일한 로직으로 인증을 구현하기 매우 편리합니다.
        
        | **장점** | **상세 내용** |
        | --- | --- |
        | **보안 (CSRF 방어)** | 자동으로 전송되지 않으므로 위조 요청 공격이 불가능함 |
        | **효율성 (선택 전송)** | 인증이 필요한 요청에만 포함하여 네트워크 낭비 방지 |
        | **유연성 (CORS/플랫폼)** | 도메인 제약이 적고 모바일 앱 등 다양한 환경에 적용 용이 |
    - 토큰을 클라이언트에 저장할 때 LocalStorage, SessionStorage, 메모리 저장의 장단점을 비교해주세요. 🍠
        
        ## 1. LocalStorage (로컬 스토리지)
        
        **"브라우저를 닫아도 기억해줘"**
        
        - **장점:** 브라우저를 닫았다가 다시 열어도 데이터가 유지됩니다. 따라서 "로그인 상태 유지" 기능을 구현하기 매우 편리합니다.
        - **단점:** 자바스크립트로 쉽게 접근할 수 있기 때문에 **XSS(스크립트 탈취) 공격**에 매우 취약합니다. 해커가 악성 코드를 심으면 토큰을 그대로 빼앗길 수 있습니다.
        
        ## 2. SessionStorage (세션 스토리지)
        
        **"지금 켜놓은 탭에서만 기억해줘"**
        
        - **장점:** 탭이나 윈도우 단위로 데이터가 격리됩니다. 탭을 닫으면 데이터가 자동으로 삭제되므로 로컬 스토리지보다 유효 기간 측면에서 조금 더 안전할 수 있습니다.
        - **단점:** 새 탭을 열거나 브라우저를 닫으면 로그인이 풀려버려 사용자 경험(UX)이 떨어질 수 있습니다. 로컬 스토리지와 마찬가지로 자바스크립트 접근이 가능하여 **XSS 공격**에는 여전히 무방비합니다.
        
        ## 3. 메모리 저장 (In-Memory)
        
        **"변수에 잠깐 담아두고 기억해줘"**
        
        - **장점:** `const token = ...` 처럼 자바스크립트 변수에 저장하는 방식입니다. 브라우저 저장소(Storage)에 아예 기록을 남기지 않으므로, 외부 스크립트가 로컬/세션 스토리지를 뒤져도 토큰을 찾을 수 없습니다. 보안상 가장 권장되는 방식 중 하나입니다.
        - **단점:** 페이지를 **새로고침(F5)** 하거나 페이지를 벗어나면 변수가 초기화되어 토큰이 사라집니다. 이를 보완하기 위해 보통 **Silent Refresh**(AccessToken이 사라지면 RefreshToken으로 몰래 다시 받아오는 기술)를 함께 사용해야 하므로 구현이 복잡합니다.
        
        | **구분** | **LocalStorage** | **SessionStorage** | **메모리 (In-Memory)** |
        | --- | --- | --- | --- |
        | **지속성** | 브라우저 종료 후에도 유지 | 탭 닫기 전까지 유지 | 새로고침 전까지만 유지 |
        | **XSS 보안** | **취약** (스크립트 접근 가능) | **취약** (스크립트 접근 가능) | **비교적 안전** |
        | **사용 편의성** | 매우 높음 | 보통 | 낮음 (추가 로직 필요) |
        | **주요 용도** | 자동 로그인 등 | 임시 데이터 보관 | 보안이 중요한 AccessToken |
    - Single Page Application, Mobile Application, Server Side Rendering Application 에서 여러분들이 생각하는 적합한 인증 전략은 무엇이라고 생각하시나요? 🍠
        
        ## 1. Single Page Application (SPA) - 예: React, Vue
        
        **추천 전략: 하이브리드 토큰 방식 (Memory + HttpOnly Cookie)**
        
        - **방식:** * **AccessToken:** 자바스크립트 메모리(변수)에 저장하여 XSS 공격으로부터 보호합니다.
            - **RefreshToken:** **HttpOnly & Secure 쿠키**에 저장하여 CSRF 공격 위험을 낮추고 보안성을 높입니다.
        - **이유:** SPA는 자바스크립트 비중이 높아 XSS 공격에 노출되기 쉽습니다. 따라서 민감한 토큰을 스토리지에 직접 노출하지 않고, 페이지가 새로고침될 때마다 쿠키에 담긴 RefreshToken으로 새로운 AccessToken을 받아오는 **Silent Refresh** 방식이 가장 안전합니다.
        
        ---
        
        ## 2. Mobile Application (Native) - 예: iOS, Android
        
        **추천 전략: 헤더 기반 토큰 방식 (Authorization Header)**
        
        - **방식:** **Authorization: Bearer <JWT>** 헤더를 사용하며, 토큰은 기기 내의 보안 저장소(iOS의 Keychain, Android의 EncryptedSharedPreferences)에 보관합니다.
        - **이유:** 모바일 앱 환경은 브라우저와 달리 **쿠키 관리나 CSRF 공격으로부터 상대적으로 자유롭습니다.** 또한, 여러 API 서버와 통신할 때 헤더 방식이 훨씬 직관적이며, 기기 자체의 강력한 암호화 저장소를 활용할 수 있어 매우 안전합니다.
        
        ---
        
        ## 3. Server Side Rendering (SSR) - 예: Next.js, JSP
        
        **추천 전략: 전통적인 세션 방식 (Session) 또는 쿠키 기반 JWT**
        
        - **방식:** 서버에서 직접 HTML을 그려주기 때문에, 브라우저와 서버가 신분을 바로 확인할 수 있는 **쿠키**를 적극 활용합니다.
        - **이유:** SSR은 첫 페이지 로딩 시 서버가 이미 사용자가 누구인지 알아야 합니다. 자바스크립트가 실행되기 전에도 인증 정보를 확인해야 하므로, 요청 시 자동으로 전송되는 **쿠키 기반**이 가장 적합합니다. 서버가 상태를 직접 관리할 수 있다면 보안성이 높은 **세션** 방식을, 서버 확장이 중요하다면 **JWT를 담은 쿠키** 방식을 선택합니다.
        
        | **환경** | **추천 방식** | **핵심 이유** |
        | --- | --- | --- |
        | **SPA** | **AccessToken(메모리) + RefreshToken(쿠키)** | XSS 방어 및 보안 강화 |
        | **Mobile** | **Authorization 헤더 + 기기 보안 저장소** | 플랫폼 호환성 및 기기 보안 활용 |
        | **SSR** | **쿠키 기반 인증 (세션 또는 JWT)** | 서버 초기 렌더링 시 인증 정보 필요 |

        ### **이유: 데이터 도청(Sniffing)의 위험**
        
        만약 비밀번호를 평문(Plain Text) 그대로 서버에 보낸다면, 사용자와 서버 사이의 통신 통로 어딘가에서 해커가 데이터를 가로챌 수 있습니다. 이를 **스니핑(Sniffing)** 공격이라고 하는데, 가로챈 데이터 안에 비밀번호가 그대로 노출되어 있다면 해커는 유저의 계정을 손쉽게 탈취하게 됩니다.
        
        ---
        
        ### **이를 방지하기 위한 핵심 기술: HTTPS (SSL/TLS)**
        
        네트워크 전송 과정에서 비밀번호를 안전하게 보호하기 위해 사용하는 핵심 기술은 바로 **HTTPS**입니다.
        
        1. **데이터 암호화:** HTTPS는 데이터를 보내기 전, 복잡한 알고리즘을 사용해 데이터를 암호화합니다. 해커가 중간에서 가로채더라도 암호화된 코드로만 보이기 때문에 실제 비밀번호가 무엇인지 알아낼 수 없습니다.
        2. **무결성 확인:** 전송 중에 데이터가 변조되지 않았는지 확인하여 안전하게 도착했음을 보장합니다.
        3. **서버 인증:** 사용자가 접속한 서버가 진짜 '아이두(Aido)' 서버가 맞는지 디지털 인증서를 통해 확인합니다.
        
        ---
        
        ### **💡 HTTP vs HTTPS 차이점 한 줄 요약**
        
        - **HTTP (HyperText Transfer Protocol):** 데이터를 암호화하지 않고 생으로 전달합니다. 마치 내용을 누구나 볼 수 있는 **'엽서'**를 보내는 것과 같습니다.
        - **HTTPS (HTTP Secure):** 데이터를 튼튼한 금고에 넣어 암호화한 뒤 전달합니다. 받는 사람만 열 수 있는 **'비밀 금고 택배'**를 보내는 것과 같습니다.
        
        ✅ 요약 : 비밀번호를 평문으로 전송하면 중간에 데이터를 가로채는 스니핑 공격에 무방비하게 노출됩니다. 이를 방지하기 위해 서버와 클라이언트는 HTTPS(SSL/TLS) 프로토콜을 사용합니다. HTTPS는 전송되는 모든 데이터를 암호화하여 설령 데이터가 유출되더라도 해커가 내용을 해독할 수 없도록 보호하는 역할을 합니다
        
        </aside>
